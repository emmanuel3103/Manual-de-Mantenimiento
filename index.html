Resumen

El presente documento constituye el Manual de Mantenimiento del sistema EDI-301, detallando los procedimientos de mantenimiento preventivo y correctivo, el plan de actualizaciones, la gestión de errores y las estrategias de respaldo y recuperación. Su finalidad es asegurar la continuidad operativa y la evolución estable del sistema.

Propósito

El propósito de este manual es establecer las directrices formales para el mantenimiento del sistema EDI-301, asegurando que las tareas preventivas, correctivas, perfectivas y adaptativas se lleven a cabo de manera ordenada, documentada y eficiente. Este documento sirve como guía para el equipo técnico responsable de mantener la estabilidad, seguridad y evolución del software, proporcionando procedimientos claros y estandarizados para la gestión de incidentes, actualizaciones y recuperación del sistema.

Alcance

El presente manual abarca las actividades esenciales de mantenimiento del sistema EDI-301, incluyendo: procedimientos de mantenimiento, manejo de errores, políticas de respaldo, recuperación ante fallos, estrategias de actualización y control de versiones. Está dirigido exclusivamente al personal técnico, administradores del sistema y desarrolladores del proyecto. No cubre aspectos funcionales del sistema (incluidos en el Manual de Usuario) ni instrucciones detalladas de desarrollo o arquitectura (tratadas en el Documento de Diseño Técnico).

1. Procedimientos de mantenimiento

1.1 Tipos de mantenimiento

  • Correctivo: resolución de errores detectados durante la operación.
  • Preventivo: revisión de rendimiento, seguridad y dependencias.
  • Perfectivo: mejoras, optimización y refactorización.
  • Adaptativo: cambios ante nuevas tecnologías o sistemas institucionales.

1.2 Procedimiento general

  1. Registrar incidente en GitHub.
  2. Clasificar prioridad y tipo de mantenimiento.
  3. Planificar y ejecutar pruebas previas.
  4. Aplicar corrección o mejora.
  5. Realizar pruebas unitarias e integradas.
  6. Desplegar en producción con versionamiento.
  7. Documentar cambios y actualizar bitácoras.

1.3 Frecuencia de revisión recomendada

  • Logs del servidor: semanal.
  • Dependencias Node.js y Flutter: trimestral.
  • Auditoría de seguridad y roles: semestral.
  • Respaldo y recuperación: semanal.

2. Plan de actualizaciones

2.1 Objetivo

Garantizar la continuidad evolutiva del sistema manteniendo un alto nivel de seguridad, compatibilidad y rendimiento.

2.2 Estrategia general

  • Uso de control de versiones Git.
  • Ramas diferenciadas (main, dev, feature/).
  • Documentación de cambios en CHANGELOG.md.
  • Pruebas automáticas previas al despliegue.
  • Comunicación con usuarios ante cambios relevantes.

2.3 Cronograma sugerido

Periodo Actividad principal Responsable
Mensual Revisión de dependencias y librerías Equipo técnico
Trimestral Actualización de API REST y módulos Flutter Desarrolladores backend / frontend
Semestral Evaluación de rendimiento y auditoría de seguridad Administrador del sistema
Anual Revisión integral de arquitectura y documentación Coordinador del proyecto

3. Manejo de errores

3.1 Registro y monitoreo

  • Captura de excepciones mediante middleware.
  • Registro en logs (archivo o plataformas externas).
  • Notificación automática de fallos críticos.

3.2 Estructura del mensaje de error

{
  "ok": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "El campo 'email' es obligatorio",
    "details": ["email"]
  },
  "timestamp": "2025-10-28T12:00:00Z"
}

3.3 Tipos de error comunes

  • 400 — Bad Request
  • 401 — Unauthorized
  • 403 — Forbidden
  • 404 — Not Found
  • 409 — Conflict
  • 500 — Internal Server Error

3.4 Procedimiento ante errores críticos

  1. Identificar módulo y reproducir error.
  2. Revisar logs y revisar versión desplegada.
  3. Aplicar hotfix inmediato.
  4. Implementar solución permanente.
  5. Notificar a responsables.

4. Copias de seguridad y recuperación

4.1 Estrategia de respaldo

  • Respaldo diario de base de datos SQL Server.
  • Respaldo semanal de archivos estáticos.
  • Sincronización constante del código en Git.
  • Backups externos en servicios en la nube.

4.2 Plan de recuperación

  1. Detener servicio afectado.
  2. Restaurar copia válida.
  3. Validar integridad.
  4. Registrar evento en bitácora.
  5. Informar restablecimiento.

4.3 Pruebas de recuperación

Ejecutar simulaciones de recuperación al menos cada semestre para garantizar la viabilidad de los respaldos y la correcta ejecución del proceso.

Conclusión

Este manual define las pautas necesarias para asegurar la operación continua del sistema EDI 301, manteniendo su seguridad, estabilidad y evolución. Su aplicación adecuada garantiza la confiabilidad del sistema dentro del entorno institucional.
